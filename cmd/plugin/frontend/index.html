Plugin Authentication[?]

This plugin needs to prove its own identity. The identity am://workload/yoyodyne/id-plugin is set up for this in the demo universe with no credentials required. If you add an ssh-pubkey annotation on that workload, you will need to get a credential via ssh and put it here.

An interesting experiment is to change the plugin's identity to am://user/the-operator. This identity does not have the attributes necessary to vouch for yoyodyne users so the login will fail even though this is a valid identity.

You can enter a username and password here. Clicking on the "Login" button will cause this plugin to validate they username and password against the sample users on the right. If this passes, then the plugin will contact the access manager to get a credential for the user.

Getting the credential will only succeed if the plugin credential below is valid and the plugin identity has permission to vouch for the user in question.

Note that the access manager knows nothing about how the plugin decides whether it should vouch for some user. The plugin knows and that is sufficient. All the access manager knows is that there is a URL for each identity. There might even be multiple plugins that can vouch for the same access manager identity. That makes migration between identity management systems much easier.

This is a list of the users that this demo plugin will accept. Obviously, in a real system, the users would be managed with a real identity management system like Active Directory or LDAP and the authentication would be more advanced than fixed passwords.

Clicking on the "Fill" button to the right of each long will fill in the username and password fields with the corresponding values.

Here you can insert a credential for the plugin itself. As the demo universe is set up, the plugin identity is not secured so we can just insert the URL for the identity directly. If we secure the identity by, for instance, adding an ssh public key as an annotation, then we would have to put a full access manager credential here instead of just a URL.

You can also experiment with putting other identities in here such as the system operator identity. Since the system operator has no visibility into the Yoyodyne user directory (under am://user/yoyodyne) login operations with the plugin using that identity will fail. This illustrates how different organizations with different identity authentication can share a single access manager.
