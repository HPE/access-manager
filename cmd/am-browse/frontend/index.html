Paste a credential here. If the identity is secured, then you will need to get a credential from the access manager. This can be done by using the id plugin example or, if the user has an ssh public key, then you can ssh to the access manager and get a credential that way.

If the identity is not secured (has no VouchFor access control and no ssh public key) then you can just enter the URL of the user or workload as a valid credential. Such an unsecured identity is only used for demos and would not be found in a production system.

Once you press the "Load Root" button, this area will be populated with metadata from the access manager. This information is displayed in the form of directories that you can open by clicking on them. Each directory or object in the system can have annotations which are shown when the object is opened if they exist. There are three kinds of annotations: a) applied roles, b) access controls, and c) user annotations. Applied roles are only used on users and workloads. The most common use of user annotations is for storing an ssh public key which can be used to authenticate a credential request.

Note that different users can see different parts of the metadata on the access manager. The user "am://user/the-operator" is the only user who can manipulate the top-level directories ("am://", "am://data", "am://role", "am://user", and "am://workload) and their content. For the purposes of this demo, there is another special user called "am://user/demo-god" who has the attribute "am://role/panopticon". All metadata on the server is configured to allow a user with that attribute to have "View" permission. This is useful during demos to show all of the metdata that exists.

The identity plugin demonstration is configured allow the "am://workload/yoyodyne/id-plugin" identity to vouch for all of the Yoyodyne users. The plugin identity is not secured to simplify the flow during demonstrations.
